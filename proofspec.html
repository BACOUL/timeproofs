Protocol (ProofSpec)

Defines how to create, sign, timestamp, and verify proofs that a specific data artifact existed at a point in time and has not changed since, without exposing the original content.

1) Abstract

ProofSpec binds a content-derived SHA-256 hash to a signed timestamp and a verification method. Transport-agnostic, publicly verifiable. Original content stays local.

2) Model & Terms

Actors

  • Author: computes the hash locally.
  • Timestamping Service: signs tuple (hash, ts) and returns a proof.
  • Verifier: checks integrity and signature.

Core flow

  1. Hash input M with SHA-256 ‚Üí hash (64-hex).
  2. POST hash (+ optional meta) ‚Üí receive signed proof.
  3. Verify signature and equality; apply time policy on ts if needed.

3) Data Model

5) Algorithms

5.1 Hashing

SHA-256 (UTF-8). Emit lowercase 64-hex.

5.2 Signing

HMAC-SHA256 with service key referenced by kid. Message:

message = `${hash}|${ts}`

5.3 Verification

  1. Recompute hash from input; compare to proof.
  2. Verify sig against the key policy for kid.
  3. Validate spec/alg and timestamp policy.

Planned: add Ed25519 profile in proofspec/1.1 (non-breaking enum extension).

6) Security & Privacy

  • Hash-only: content never leaves the device.
  • No personal data in meta.
  • Keys: rotation & revocation policy; bind via kid.
  • Clock: multi-source NTP; tolerance ‚â§ 5 s.
  • Abuse: per-IP & per-key rate limiting; optional nonce.

10) Reference Implementation

Create (cURL)

curl -sS -X POST https://api.timeproofs.io/api/timestamp \
  -H "Content-Type: application/json" \
  --data '{"hash":"<64-hex>","type":"event","meta":{"source":"spec"}}'

Verify (cURL)

curl -sS "https://api.timeproofs.io/api/verify?hash=<64-hex>"

JavaScript (fetch)

// Create
const resp = await fetch("https://api.timeproofs.io/api/timestamp",{
  method:"POST",
  headers:{ "Content-Type":"application/json","Accept":"application/json" },
  body: JSON.stringify({ hash:"<64-hex>", type:"event", meta:{ source:"spec" } })
});
const proof = await resp.json();

  headers:{ "Accept":"application/json" }

12) FAQ

Is blockchain required?

No. Standard hashing + signatures. Lower cost, higher privacy.

What does the timestamp guarantee?

That the hash existed at or before the recorded time and matches unchanged content.

What goes in meta?

Optional context (e.g., type, source). Do not include personal data.

How are keys rotated?

Each proof references a kid; rotation policy is published per kid.

Can proofs be exported?

Yes ‚Äî .tproof.json bundles for offline verification.
