ProofSpec — TimeProofs Open Protocol for Digital Proof of Existence

Defines how to create, sign, timestamp, and verify proofs that a specific data artifact existed at a point in time and has not changed since, without exposing the original content.

1. Abstract

ProofSpec defines a minimal, interoperable protocol to bind a content-derived hash to a signed timestamp and a verification method. The protocol is transport-agnostic and verifiable by anyone. Original content never leaves the device.

2. Model & Terms

Actors

  • Author: computes the hash locally.
  • Timestamping Service: signs a tuple (hash, time, meta).
  • Verifier: checks integrity and signature.

Core flow

  1. Hash input M with SHA-256 → hash.
  2. Submit hash + optional meta → receive signed Proof.
  3. Verify signature, time window, and equality of hashes.

3. Data Model

{
  "spec": "proofspec/1.0",
  "hash": "sha256:...hex...",
  "alg":  "hmac-sha256",
  "ts":   "2025-10-19T14:00:00Z",
  "meta": { "type":"event", "source":"api", "nonce":"base64url" },
  "sig":  "base64url(hmac(secret, hash || ts || meta))",
  "kid":  "key-id-or-issuer",
  "ver":  { "endpoint":"https://.../api/verify", "method":"GET" }
}

Fields are stable and self-describing. meta is optional and must exclude personal data.

5. Algorithms

5.1 Hashing

Use SHA-256. Text must be UTF-8 encoded. Binary inputs hashed directly.

5.2 Signing

Default: HMAC-SHA256 with a service key identified by kid. Message is the UTF-8 concatenation:

message = hash + "\n" + ts + "\n" + base64url(JSON.stringify(meta || {}))

5.3 Verification

  1. Recompute hash from user input; compare.
  2. Rebuild message; verify sig using key for kid.
  3. Check ts freshness vs. policy; validate alg and spec.

5.4 Transport

Out of scope. HTTP(S) reference endpoint provided in ver.endpoint.

6. Security & Privacy

  • Hash-only: original content never transmitted.
  • Personal data: must not be placed in meta.
  • Keys: rotate; bind kid to rotation policy.
  • Replay: use nonce in meta if needed.
  • Time: service clocks must be NTP-synchronized.

7. Regulatory Mapping

EU AI Act / Data Act / DSA

Supports traceability logs, integrity checks, and auditability of generated or processed content.

GDPR

No personal data required. Proofs are hashes+metadata; keep meta non-identifying.

ProofSpec complements C2PA by focusing on existence + integrity, not provenance chains.

8. Versioning

  • spec is proofspec/MAJOR.MINOR. Backward compatible within MINOR.
  • Breaking changes increment MAJOR.
  • Reference implementation is tagged with matching protocol versions.

11. FAQ

Is blockchain required?

No. ProofSpec uses standard hashing and signatures. Lower cost and higher privacy.

Can I verify offline?

Yes, if you possess the proof object and the public key or shared key policy for kid.
