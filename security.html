Security

Our security posture: hash-only data model, strong transport security, protective headers, key hygiene, backups, SBOM, disclosure policy, and audit roadmap.

1) Threat Model

  • Assets: timestamp proofs, API keys, signing keys, build artifacts, documentation.
  • Adversaries: network attackers, key thieves, malicious clients, supply-chain compromise.
  • Goals: prevent tampering, key leakage, and unauthorized API use; ensure verifiability.

2) Data Handling

  • Hash-only model: customers hash locally; originals never sent to TimeProofs.
  • Stateless API: requests are independent; minimal logs (timings, status codes).
  • PII/PHI: not required; customers must avoid personal data in meta.
  • Retention: logs with bounded retention; no content storage by default.

3) Transport & Security Headers

  • TLS: HTTPS everywhere; HSTS enabled at the edge.
  • CSP: restrictive default + worker-src 'self' blob: to keep SW.
  • Other headers: COOP/COEP considered where applicable; Referrer-Policy strict; X-Content-Type-Options nosniff.
  • Service Worker: registered as module at /sw.js.

4) Key Management

  • Server secrets: scoped and rotated; no long-lived master tokens in code.
  • Signing: HMAC-SHA256 in v0.1; Ed25519 planned with JWKS at /.well-known/jwks.json (key rotation via kid).
  • Client API keys: per-tenant tokens; rate limits per key and per IP.

5) Backups & Recovery

  • KV backups: periodic snapshots + test restores (runbook documented).
  • Disaster recovery: multi-region edge; runbook for provider switch (BYOI support).

6) Dependencies & SBOM

  • Dependency audit: automated updates (Dependabot) and vulnerability scans.
  • SBOM: generated per release; retained with build artifacts.

7) Vulnerability Disclosure

  • Policy: see /.well-known/security.txt.
  • Contact: contact@timeproofs.io (PGP optional in roadmap).
  • Safe harbor: good-faith testing honored; avoid impacting other users.

8) Availability & Monitoring

  • SLO (target): 99.9% monthly; latency budget tracked.
  • Monitoring: edge health checks, error rates, saturation, regional latency.
  • Status page: on roadmap with transparency reports.

9) Audits & Penetration Tests

  • External audit: planned for post-v1.0.
  • Pentests: scoped before major releases; remediation tracked publicly when possible.
